# Claude Code 社内紹介プロジェクト - Claude Rules

## 📋 プロジェクト概要

このプロジェクトはClaude Codeを社内に紹介するための資料作成を目的としています。
効果的なプレゼンテーション資料と実践的なデモを通じて、Claude Codeの価値を伝えます。

---

## 🎯 プロジェクトの目的

1. Claude Codeの機能と特徴を明確に理解する
2. 実際のユースケースを具体的に示す資料を作成する
3. 実演可能なデモコードを準備する
4. 社内の開発者に導入メリットを効果的に伝える

---

## 📁 ディレクトリ構造とルール

### ディレクトリの役割

```
.
├── 01_reference/           # 公式ドキュメント、参考資料（読み取り専用）
├── 02_templates/           # テンプレートファイル
├── 03_my_implementations/  # デモコード・サンプル実装（書き込み可）
├── 04_notes/               # 調査ノート・アイデアメモ（書き込み可）
├── 05_progress/            # 進捗管理（書き込み可）
└── 06_docs/                # 最終的な社内紹介資料（書き込み可）
```

### 重要なルール

1. **`01_reference/` は読み取り専用**
   - 公式ドキュメントや参考資料は変更しない
   - 調査や理解のためにのみ参照する

2. **`03_my_implementations/` にデモコードを実装**
   - 実演用のサンプルコードはここに配置
   - トピックごとにサブディレクトリを作成
   - **ディレクトリ命名規則**: `{NN}_{topic_name}/` (例: `01_basic_demo/`, `02_tdd_workflow/`)
   - **連番の接頭辞**: 理解しやすい順序で連番（01, 02, 03...）を接頭辞として付ける
   - **重要**: Pythonファイル名には連番を付けない（importエラーになるため）
     - ✅ ディレクトリ: `01_basic_demo/`, `02_tdd_workflow/`
     - ❌ Pythonファイル: `01_calculator.py`（数字で始まるとimportできない）
     - ✅ Pythonファイル: `calculator.py`, `data_processor.py`

3. **`04_notes/` に調査メモを記録**
   - 機能調査、アイデア、構成案をまとめる
   - **命名規則**: `{NN}_{topic_name}.md`（連番接頭辞を必ず付ける）
   - **連番の目的**: 調査の流れを明確にする

4. **`05_progress/` で進捗を管理**
   - `learning_log.md` は**進捗の把握のみに使用**
     - どこまで完了したか（チェックリスト）
     - 次にどこから始めるか（次のタスク）
   - **詳細な内容は各ディレクトリのREADMEで確認する**
     - 実装の詳細、技術的な内容は各タスクのREADME.mdに記載
     - learning_log.mdには詳細を書かず、完了日とステータスのみ記録

5. **`06_docs/` に最終資料を配置**
   - 社内向けプレゼンテーション資料
   - デモ手順書
   - FAQ資料

## 💻 コーディング規約

### Python（デモコード用）

- **バージョン**: Python 3.13以上
- **パッケージマネージャー**: uv
- **スタイル**: PEP 8に準拠
- **型ヒント**: 可能な限り使用する
- **docstring**: 重要な関数・クラスには必須
- **コメントとdocstringの言語**: **必ず日本語で記載すること**
  - ✅ モジュールdocstring: 日本語
  - ✅ 関数/クラスdocstring: 日本語
  - ✅ インラインコメント: 日本語
  - ❌ 英語のコメントは使用しない
- **命名規則**:
  - 関数/変数: `snake_case`
  - クラス: `PascalCase`
  - 定数: `UPPER_CASE`

### Markdown（資料作成用）

- **見出し**: 階層を明確にする
- **コードブロック**: 言語指定を必ず行う
- **リンク**: 相対パスを使用
- **画像**: `06_docs/images/` に配置

## 📝 資料作成プロセス

### 作成フロー（全7フェーズ）

#### 1. **調査フェーズ** 📖
   - Claude Code公式ドキュメントを読む
   - 既存の紹介資料やブログ記事を調査
   - ユースケースや事例を収集
   - `04_notes/` にメモを作成

#### 2. **構成設計フェーズ** 📋
   - 資料の全体構成を設計
   - ターゲットオーディエンスを明確化
   - 伝えるべきメッセージを整理
   - ストーリーラインを作成

#### 3. **コンテンツ作成フェーズ** ✍️
   - 各セクションのコンテンツを作成
   - 図表やスクリーンショットを準備
   - わかりやすい説明文を書く
   - `06_docs/` に資料を配置

#### 4. **デモ準備フェーズ** 💻
   - 実演用のサンプルコードを作成
   - デモシナリオを設計
   - `03_my_implementations/` に実装
   - 動作確認とリハーサル

#### 5. **レビューフェーズ** 🔍
   - 資料の内容を確認
   - デモの流れをチェック
   - フィードバックを収集
   - 改善点を洗い出し

#### 6. **プレゼン準備フェーズ** 🎤
   - スライド資料を完成させる
   - デモ環境を準備
   - Q&A想定問答を作成
   - リハーサルを実施

#### 7. **振り返りフェーズ** 📊
   - **ドキュメント更新（必須）**
     1. 各タスクディレクトリの `README.md` を更新
     2. プロジェクトルートの `README.md` を更新（全体の進捗を反映）
     3. `progress/learning_log.md` に完了日とステータスを記録
   - **成果物の整理**
     - 最終資料を `06_docs/` に配置
     - デモコードの動作を最終確認
   - **Git操作**
     - 変更をコミット
     - プッシュ（Claude Codeは実行前に必ず確認を取る）

## 🔧 資料作成時の注意事項

### セキュリティ

- **機密情報**: 社内の機密情報は含めない
- **個人情報**: 実名やメールアドレスは使用しない
- **APIキー**: デモで使用するキーは環境変数で管理

### 対象オーディエンス

- **技術レベル**: 開発者向け
- **前提知識**: AIツールの基本的な理解
- **興味関心**: 生産性向上、コード品質改善

### プレゼンテーション

- **時間配分**: 全体で30-45分程度
- **デモ時間**: 15-20分程度を確保
- **質疑応答**: 10分程度を想定

## 📊 進捗管理

### progress/learning_log.md の役割

**重要**: `progress/learning_log.md` は**進捗把握専用**です。詳細な内容は書きません。

**記載内容**:
- ✅ どこまで完了したか（チェックリスト）
- ✅ 次にどこから始めるか（次のタスク）
- ✅ 完了日のみ（例: `2025-11-05`）
- ❌ 資料の詳細内容（各READMEで確認）
- ❌ 調査内容の詳細（04_notes/で確認）

### ドキュメント更新フロー（必須）

**各タスク完了後は必ず以下の順序でドキュメントを更新すること**:

1. **タスクのREADME.md**
   - タスクの詳細、成果物、学んだことを記載
   - 他の人が見てもわかるように丁寧に書く

2. **プロジェクトルートのREADME.md**
   - 全体の完了タスク数を更新
   - 完了したタスクのリストを更新

3. **progress/learning_log.md**
   - 完了日とステータスのみ記録
   - 詳細は書かない

## 🤝 Claude Codeとの協働ルール

### Claude Code に期待すること

1. **資料構成支援**: 効果的な資料構成の提案
2. **コンテンツ作成**: わかりやすい説明文の作成補助
3. **デモコード作成**: 実演用サンプルコードの実装
4. **レビュー**: 資料とコードの改善提案
5. **FAQ作成**: よくある質問と回答の作成補助

### Claude Code が守るべきこと

1. **段階的作成**: 一度に全てを作成せず、小さなステップで進める
2. **わかりやすさ重視**: 専門用語を避け、平易な表現を使う
3. **具体性**: 抽象的な説明ではなく、具体例を示す
4. **実践的**: 実際に使える情報を提供する
5. **Git操作の確認**: `git push` は実行前に必ずユーザーの確認を取る
6. **ドキュメント更新**: 各タスク完了後、**必ず3段階のドキュメント更新**を実施する
   - タスクのREADME.md → プロジェクトのREADME.md → learning_log.md
7. **命名規則**:
   - ディレクトリには連番を接頭辞として付ける（01_, 02_, 03_...）
   - Pythonファイル名には連番を付けない（importエラーを回避）

## 🔒 個人情報とセキュリティルール

### 個人情報の取り扱い

このプロジェクトは社内向けですが、将来的に公開する可能性があるため、個人情報の管理を徹底してください。

#### 禁止事項

以下の個人情報をコード、ドキュメント、コミットメッセージに含めないでください：

1. **実名**
2. **連絡先情報**（メールアドレス、電話番号）
3. **社内固有情報**（社員番号、部署名、プロジェクト名）

#### 推奨事項

**匿名化を徹底**:
- 例示には架空の名前や情報を使用
- スクリーンショットには個人情報を含めない
- デモコードでは一般的な例を使用

### 機密情報の取り扱い

#### 管理が必要な情報

1. **APIキー・トークン**
2. **環境変数**
3. **秘密鍵**

#### 安全な管理方法

**環境変数ファイル**:
```bash
# .env ファイルを使用（gitignore必須）
API_KEY=your_key_here
```

**サンプルファイルの提供**:
```bash
# .env.example をリポジトリにコミット
API_KEY=your_key_here
```

**.gitignoreの設定**:
```gitignore
# Secrets
*.pem
*.key
credentials.json

# MCP Configuration
.mcp.json

# Environment variables
.env
.env.local
```
